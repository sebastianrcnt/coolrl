# 오목 AI 약화 메커니즘 최종 제안

## 한 줄 요약

**현재 단순 τ 다이얼은 폐기하고, "그럴듯한 후보군을 먼저 추리고 그 안에서만 부드럽게 흔드는" 방식으로 교체한다.**

---

## 왜 지금이 망가져 있나

현재 `τ > 0`이면 모든 합법수 N(s,a)에 `N^(1/τ)` 샘플링을 그대로 적용한다.

15×15 오목은 합법수가 200개 가까이 되고 MCTS는 96~128회만 돌리니까 방문수 분포가 이렇게 생긴다.

```
베스트 수:        60 방문
2~5위:            5~20 방문
잡수 50+개:       1~2 방문씩 (탐색이 한 번씩 찔러본 것)
```

τ=1을 그대로 걸면 잡수 꼬리(합 ~70)가 베스트(60)를 압도한다. 사삼 막아야 할 자리에서도 70% 확률로 다른 데를 둔다. **약화가 아니라 탐색 노이즈를 착수로 승격시키는 동작**이 돼 있다.

알파제로 논문의 τ=1은 자가대국 *학습 데이터 다양성* 용도이지 추론용이 아니다. 그걸 그대로 가져온 게 잘못이었다.

---

## 핵심 원칙

> **약화는 후보군을 넓히는 것이지, 후보도 아닌 수를 살려내는 게 아니다.**

사용자가 "AI가 멍청하게 뒀네"가 아니라 "AI가 최선수를 놓쳤네"로 받아들여야 효능감이 산다. 그래서 약화는 항상 **상위 후보 안에서만** 일어나야 한다.

---

## 제안 구조: Filtered Softmax

다섯 단계로 나눈다.

```
1. 후보군 추리기
   ├─ visit ratio 컷       (잡수 꼬리 제거)
   ├─ top-K 컷             (후보 수 상한)
   └─ Q 블런더 컷          (명백한 자살수 제거)

2. 후보가 1개면 그대로 둔다 (강제수 보호)

3. 후보 안에서만 softmax 샘플링
   score = log(N+1) + qWeight·Q + priorWeight·log(P)
   π(a) ∝ exp(score / τ)
```

각 단계의 역할:

| 단계 | 막는 것 |
|---|---|
| visit ratio 컷 | 1~2방문짜리 잡수 꼬리 |
| top-K 컷 | 후보가 너무 많아지는 상황 방지 |
| Q 블런더 컷 | "Q가 매우 나쁜데 우연히 visit이 좀 있는 수" |
| 후보 1개 보호 | 진짜 강제수(사삼 방어 등) 안전장치 |
| 후보 내 softmax | 자연스러운 흔들림 |

## 다이얼: 단일 weakness 값 (0~1)

내부 파라미터가 5개라 사용자에게 직접 노출은 부담. **`weakness ∈ [0, 1]` 한 값**으로 압축하고 모든 파라미터는 보간으로 결정한다.

```ts
function paramsFromWeakness(w: number) {
  return {
    temperature:    lerp(0.0,  1.05, w),
    topK:           Math.round(lerp(1,    12,   w)),
    minVisitRatio:  lerp(1.0,  0.09, w),
    qDrop:          lerp(0.0,  0.38, w),
    qWeight:        lerp(3.5,  1.5,  w),
    priorWeight:    lerp(0.1,  0.35, w),
  };
}
```

`weakness = 0`일 때는 그냥 argmax로 단락(short-circuit). 보간 보다 명시적인 게 안전.

향후 게임 결과 EMA로 자동 조정할 때도 weakness 한 값만 움직이면 된다.

## UI: 5단계 프리셋

| 라벨 | weakness | 의도 |
|---|---|---|
| **강함** | 0.00 | 거의 최선수만 (현재 τ=0과 동일) |
| **어려움** | 0.25 | 가끔 차선수, 방어는 확실 |
| **보통** | 0.45 | 사람 같은 흔들림, 사용자 승률 50~60% 목표 |
| **쉬움** | 0.65 | 방어는 하지만 공격이 무딤, 승률 65~75% |
| **매우 쉬움** | 0.85 | 두루 무름, 단 명백한 자살수는 X |

기본값은 **보통**. 첫 사용자에게 "이길 수 있다"는 인상이 중요하면 **쉬움**도 후보.

τ 0~2 다이얼은 폐기하고 이 5단계 select로 대체.

## 구현 위치

현재 `mcts.ts` `chooseAction`에서 root visit count argmax로 끝내는 부분을 새 선택기로 교체.

함수 시그니처:

```ts
function chooseMoveWithWeakening(
  children: Array<{ action, visits, prior, q }>,
  params: WeakeningParams,
  rng: () => number
): number
```

MCTS 자체는 안 바꾼다. 루트에서 자식 통계를 뽑은 뒤 이 함수로 액션 하나만 정하면 됨.

힌트(추천 수)는 weakness 무시하고 항상 argmax — 사용자에게 추천할 때는 최선수여야 의미가 있다.

---

## 안전장치 두 가지

### 1. Q 방향 확인 (구현 시 필수 점검)

`q`가 "AI 관점"으로 들어와야 한다. 즉 Q가 클수록 AI에게 좋은 수여야 함.

방향이 반대로 들어오면 `qDrop` 필터가 **최악의 수를 보호하는 버그**가 된다. 코드 한 번에 죽인다.

현재 MCTS의 `child.averageValue()`는 자식 노드의 to-play 관점이라 부호 뒤집기가 필요할 수 있다. 구현할 때 단위 테스트로 확인.

### 2. Dominant best move 우회

베스트가 차선보다 압도적이면(예: 60방문 vs 15방문) 그건 사실상 강제수일 가능성이 크다. 이런 경우는 weakness 값과 무관하게 argmax로 단락.

임계값은 **sims 비례**로 잡는다. 절대값으로 두면 sims=96과 sims=256에서 의미가 달라짐.

```ts
function hasDominantBestMove(legal, sims): boolean {
  const [best, second] = sorted;
  const visitDominance =
    best.visits >= second.visits * 3.0 &&
    best.visits >= 0.20 * sims;        // sims 96→19, 256→51
  const qDominance =
    best.q >= second.q + 0.25 &&
    best.visits >= 0.13 * sims;        // sims 96→12, 256→33
  return visitDominance || qDominance;
}
```

단, 이걸 너무 강하게 켜면 쉬움 난이도에서도 너무 강해진다. **보통 이상 난이도에만 적용**.

## 기존 단순안 대비 우월한 점

이전에 검토한 4가지 후보(top-K만 / min-visit ratio만 / Q softmax만 / top-p만) 어느 하나만 써도 부족했다.

| 단독 방식 | 약점 |
|---|---|
| top-K만 | K가 고정이라 국면 적응성 X |
| min-visit ratio만 | 정책망이 잘못 쏠릴 때 좋은 수가 잘림 |
| Q softmax만 | N=1짜리 잡수의 Q 노이즈가 그대로 들어옴 |
| top-p만 | 원본 분포가 망가져 있어서 무의미 |

세 필터를 차례로 적용하면 각자 약점을 서로 보완한다. 복잡도는 늘지만 함수 하나에 들어가니 통제 가능.

## 단계별 도입 계획

작업 단위가 크니 두 단계로 쪼갠다.

### 1단계 (MVP)
- `mcts.ts`에 `chooseMoveWithWeakening` 함수 추가
- `RunOptions.temperature` → `RunOptions.weakness` 로 교체
- UI 다이얼을 5단계 select로 교체
- 기본값 **보통(0.45)**
- 단위 테스트: weakness=0이면 argmax, weakness=1이면 다양성 확인

### 2단계 (튜닝)
- 실제 5~10판 두면서 각 프리셋의 체감 확인
- 사용자 승률이 너무 낮으면 weakness 값 위로 시프트
- "보통에서 가끔 자살수" 발견되면 `qDrop` 강화
- 기본값을 사용자군에 맞춰 재조정

### 3단계 (자동 조정, 미래)
- 게임 종료 시 결과 EMA 추적
- weakness를 자동으로 ±0.05씩 조정
- 비대칭(이기면 천천히 강화, 지면 빠르게 약화) 적용

## 절대 피할 것

```
✗ 전체 legal move에 N^(1/τ) softmax
✗ 전체 legal move에 Q softmax
✗ K 없이 τ만 올리기
```

이 셋은 잡수 꼬리 문제를 그대로 둔 채 약화하는 척만 하는 방식이다. 현재 τ 다이얼이 첫 번째 케이스에 해당.

## 기존 sims 다이얼과의 관계

지금 설정 시트에는 이미 **탐색 횟수**(64 쉬움 / 96 중간 / 128 어려움 / 256 알파고) 가 있다. 새 weakness 다이얼과 축이 겹친다.

두 축의 차이는 다음과 같다.

| 다이얼 | 통제하는 것 | 부수효과 |
|---|---|---|
| **sims (탐색 횟수)** | AI가 얼마나 깊이 생각하는가 (compute) | 1수 사고 시간 ↑↓, 트리 통계의 신뢰도 ↑↓ |
| **weakness** | 그 생각 끝에 어떻게 고르는가 (quality) | 자연스러운 흔들림, 자살수 회피 |

**문제점**: 사용자에게 두 다이얼 다 노출하면 헷갈린다. "256 + 매우쉬움" vs "64 + 강함" 어느 게 더 약한가? 답이 자명하지 않다.

또 sims가 너무 낮으면(예: 32~64) 베스트 visit이 5~12에 머물러서 visit ratio 컷 / top-K 컷이 의미를 잃는다. 즉 weakness 메커니즘은 sims가 충분할 때(96+) 가장 잘 작동한다.

**결정**: sims는 **[96, 256] 범위로 고정**. 그 밑은 weakness 메커니즘이 깨지고, 그 위는 모바일 OOM 위험. 이 범위 안에서는 visit ratio / top-K / Q-drop 모두 sims 비례로 동작하므로 weakness 한 값만 움직여도 일관된 체감이 나온다.

### 권장: 두 축을 하나로 묶는다

사용자에게는 **난이도 5단계** 하나만 노출하고, 내부에서 두 축을 같이 움직인다.

| 난이도 | sims | weakness | 1수 사고시간 (대략) |
|---|---|---|---|
| 강함 | 256 | 0.00 | 길다 |
| 어려움 | 128 | 0.25 | 보통 |
| 보통 | 128 | 0.45 | 보통 |
| 쉬움 | 96 | 0.65 | 짧다 |
| 매우 쉬움 | 96 | 0.85 | 짧다 |

논리:
- 강할수록 sims↑, weakness↓ (둘 다 강 방향)
- 약할수록 sims↓, weakness↑ (둘 다 약 방향, 게다가 사고시간 짧아져서 답답함도 줄어듦)
- sims는 [96, 256] 범위에 고정. 64는 쓰지 않음 — weakness 메커니즘이 망가지는 영역.

### sims 다이얼은 어떻게 처리?

세 가지 안:
1. **완전 폐기** — 단순함 최대. 사용자는 난이도 하나만 본다.
2. **고급 설정 안으로 이동** — 평소엔 자동, 파워유저는 수동 오버라이드 가능.
3. **유지하되 라벨 변경** — "사고 깊이" 같은 중립 이름으로. 단 난이도 라벨과 충돌 여전.

**추천**: 1번 (완전 폐기) 또는 2번 (고급 안에 숨김). 디지털 치매 치료용이라 단순함이 최우선. 1번을 강권.
